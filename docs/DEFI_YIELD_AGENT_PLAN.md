# DeFi 수익 에이전트 — 계획서

> **버전**: v0.1 (초안) · **작성일**: 2026-07-16
> **목표**: 4개 체인(Ethereum, BSC, Solana, Base)의 TVL 상위 DeFi 프로토콜만을 활용해, AI 에이전트가 리서치·모니터링·리밸런싱을 수행하는 **전략적으로 안정적인 수익** 시스템을 구축한다.
> **핵심 철학**: 수익률 극대화가 아니라 **원금 보존 우선 + 검증된 프로토콜 + 자동화된 감시**. "지루하지만 잃지 않는" 포트폴리오.

---

## 1. 프로젝트 개요

### 1.1 왜 Top TVL 프로토콜인가

| 항목 | 근거 |
|---|---|
| 스마트컨트랙트 리스크 | TVL이 크고 오래된 프로토콜일수록 감사·실전 검증(린디 효과)이 축적됨 |
| 유동성 | 대규모 TVL = 진입/이탈 시 슬리피지 최소, 긴급 출금 가능성 높음 |
| 정보 접근성 | DeFiLlama 등 공개 API로 TVL·APY·리스크 지표를 에이전트가 실시간 조회 가능 |
| 거버넌스 안정성 | 상위 프로토콜은 급격한 파라미터 변경·러그 확률이 구조적으로 낮음 |

### 1.2 왜 에이전트인가

- DeFi 수익률은 **매일 변한다** (금리 시장). 사람이 4개 체인 × 10여 개 풀을 매일 추적하는 것은 비현실적.
- 에이전트의 역할: ① 수익률/TVL/디페그 **모니터링**, ② 리밸런싱 **제안**, ③ 리스크 이벤트 **알림**, ④ (검증 후) 제한된 범위 내 **자동 실행**.
- Hyperliquid 봇(quant)에서 만든 텔레그램 UI + 이벤트 버스 패턴을 재사용한다.

### 1.3 설계 원칙

1. **화이트리스트 원칙** — 아래 §2에서 정의한 프로토콜 외에는 에이전트가 어떤 제안도 하지 않는다. 신생 고APY 풀은 원천 차단.
2. **읽기 → 제안 → 반자동 → 자동** 4단계. 각 단계는 최소 4주 무사고 운영 후에만 승격.
3. **레버리지 금지(초기)** — 루핑, 순환 담보 대출은 Phase 3 이전 금지. 렌딩은 "예치(supply)만".
4. **브리지 최소화** — 체인 간 이동은 CEX(업비트/바이낸스) 경유 또는 네이티브 발행(USDC CCTP)만 사용. 서드파티 브리지 금지.
5. **키 분리** — 에이전트는 실행 권한이 없는 읽기 전용으로 시작. 실행 도입 시에도 세션 키/한도 지갑(스마트 계정)으로 권한 제한.

## 4. 전략 구조 — 코어/위성

### 4.1 코어 (70~80%) — 스테이블코인 렌딩

- 자산: USDC, USDT, USDS (검증된 법정화폐 담보/과담보만; 알고리즘 스테이블 제외)
- 배치: Jupiter Lend(SOL) · Morpho 큐레이터 볼트(Base) · Sky sUSDS(ETH) · Aave V3(ETH/Base)
- 규칙: 단일 프로토콜 최대 35%, 단일 체인 최대 40%, 단일 스테이블코인 최대 50%
- 목표: 연 4~6%, 무손실

### 4.2 위성 (20~30%) — LST + 선별 LP

- ETH → stETH(Lido), SOL → jitoSOL(Jito/Sanctum): 코인 개수 증가 전략 (가격 리스크는 별도 판단)
- PancakeSwap/Raydium 주요 페어 LP: **IL(비영구손실) 시뮬레이션 통과한 스테이블-스테이블 또는 상관계수 높은 페어만**
- 목표: 코인 기준 연 2~7% 추가

### 4.3 금지 목록

- 화이트리스트 외 프로토콜 전부 · 레버리지 루핑 · 서드파티 브리지 · 출시 6개월 미만 풀 · APY 15% 초과 풀(에이전트가 자동으로 "의심" 분류)

## 5. 에이전트 아키텍처

```
┌────────────────────────────────────────────────────────────┐
│                    defi-agent (단일 프로세스)                 │
│                                                            │
│  ┌───────────────┐        ┌────────────────────┐           │
│  │ Telegram Bot   │◄──────►│ Agent Core          │           │
│  │ (승인/알림 UI)  │        │ (Claude API)        │           │
│  └───────────────┘        │  - 일일 시장 브리핑   │           │
│                           │  - 리밸런싱 제안      │           │
│  ┌───────────────┐        │  - 리스크 판단        │           │
│  │ Scheduler      │───────►└─────────┬──────────┘           │
│  │ (cron: 1h/24h) │                  │                      │
│  └───────────────┘        ┌─────────▼──────────┐           │
│                           │ Data Layer          │           │
│  ┌───────────────┐        │ - DeFiLlama API     │           │
│  │ SQLite         │◄──────│ - 체인 RPC (잔고)    │           │
│  │ (포지션/이력)   │        │ - 가격 오라클        │           │
│  └───────────────┘        └─────────┬──────────┘           │
│                           ┌─────────▼──────────┐           │
│                           │ Execution Layer     │  Phase 2+ │
│                           │ (한도 지갑, 멀티체인) │           │
│                           └────────────────────┘           │
└────────────────────────────────────────────────────────────┘
```

### 5.1 에이전트의 4가지 잡

| 잡 | 주기 | 내용 |
|---|---|---|
| **모니터링** | 1시간 | TVL 급감(-15%/24h), 디페그(±0.5%), APY 급변 감지 → 즉시 텔레그램 알림 |
| **일일 브리핑** | 24시간 | 포지션 현황, 수익 집계, 체인별 금리 변동 요약 |
| **리밸런싱 제안** | 주 1회 | 금리 격차가 이동 비용(가스+슬리피지) 대비 손익분기 초과 시에만 제안 |
| **긴급 대응** | 이벤트 | 해킹 뉴스/거버넌스 공격 감지 시 "전량 출금 제안" + 사용자 승인 대기 |

### 5.2 기술 스택

| 레이어 | 선택 | 근거 |
|---|---|---|
| 언어 | Python 3.12+ | quant 봇과 동일 스택, 코드 재사용 |
| 에이전트 | Claude API (tool use) | 데이터 조회 도구를 주고 판단·요약을 위임 |
| 데이터 | DeFiLlama API (무료) + web3.py / solana-py | TVL·APY·잔고 조회 |
| UI | aiogram (텔레그램) | quant 봇 패턴 재사용 |
| 저장 | SQLite | 포지션·제안·승인 이력 |
| 실행(후순위) | Safe(EVM) 세션 키 + Solana 한도 지갑 | 에이전트 권한을 금액·프로토콜 단위로 제한 |

## 6. 리스크 관리

| 리스크 | 대응 |
|---|---|
| 스마트컨트랙트 해킹 | Top TVL 화이트리스트 + 프로토콜당 35% 상한 + 4체인 분산 |
| 스테이블 디페그 | 단일 스테이블 50% 상한, ±0.5% 이탈 시 즉시 알림, USDe 등 합성 스테이블은 별도 소분류·상한 10% |
| 브리지 해킹 | 서드파티 브리지 사용 금지 (CEX 경유 / CCTP만) |
| 에이전트 오작동 | Phase 2까지 모든 실행에 인간 승인 필수, 실행 지갑에 하드 한도 |
| 키 유출 | 읽기 전용 시작, 실행 도입 시 메인 지갑과 운용 지갑 분리 |
| 금리 하락 | 목표 수익률을 고정하지 않음 — "시장 금리 추종"이 목표 |

## 7. 로드맵

| Phase | 내용 | 승격 조건 |
|---|---|---|
| **0. 관측** (2~3주) | DeFiLlama 수집기 + SQLite + 텔레그램 일일 브리핑. 자금 투입 없음 | 브리핑 데이터 정확성 검증 |
| **1. 수동 운용** (4주+) | 소액 실제 예치(수동 실행), 에이전트는 모니터링·제안만 | 4주 무사고 + 제안 품질 확인 |
| **2. 반자동** | 에이전트 제안 → 텔레그램 버튼 승인 → 자동 실행 (한도 지갑) | 8주 무사고 |
| **3. 자동화** | 사전 정의 규칙 내 자동 리밸런싱, 승인은 사후 보고 | 별도 재평가 |

**다음 액션 (Phase 0)**:
1. `defi-agent` 프로젝트 스캐폴딩 (quant 구조 재사용)
2. DeFiLlama 수집기: 화이트리스트 프로토콜의 TVL/APY 1시간 주기 수집
3. 텔레그램 일일 브리핑 봇 (§3 스냅샷 표 자동 생성)

## 8. 예상 수익 (보수적)

- 스테이블 코어(75%): 연 4~6% → 기여 3.0~4.5%
- LST/LP 위성(25%): 연 2~7% (코인 기준) → 기여 0.5~1.8%
- **포트폴리오 전체: 연 4~6% (USD 기준, 코인 가격 변동 제외)**
- 이 수치는 2026-07 금리 스냅샷 기반이며, 에이전트가 추적하는 것은 "수치"가 아니라 "시장 대비 최적 배치"다.
